# 内容与元数据规范

Front Matter 是内容与模板之间的稳定接口。所有新文章必须遵守本规范；旧文章按实际维护节奏逐步补齐。

## 必填字段

```yaml
---
layout: post
title: "准确描述文章解决的问题"
description: "80 至 160 字的独立摘要，用于列表、搜索和社交分享"
date: 2026-06-25 10:00:00 +0800
categories: [tech]
tags: [Jekyll, GitHub Pages]
article_type: guide
content_score: 7.0
score_basis: "真实问题明确，解决路径可验证，长期可搜索；不足是适用场景较窄。"
permalink: /tech/example/
---
```

| 字段 | 规则 |
| --- | --- |
| `layout` | 文章固定为 `post`。 |
| `title` | 清晰、具体；避免只有情绪或宽泛主题。 |
| `description` | 不依赖正文也能理解；说明问题、范围或结论。 |
| `date` | 包含时区。发布日期变化时不要覆盖原日期。 |
| `categories` | 使用数组且只能有一个一级分类：`tech` 或 `tutorials`。 |
| `tags` | 使用数组；建议 2 至 5 个，采用项目既有拼写。 |
| `article_type` | 使用下方定义的固定值。 |
| `content_score` | 文章自评分，范围 6.0 至 10.0；低于 6 分不发布。 |
| `score_basis` | 一句话说明评分依据，必须能指出价值和不足。 |
| `permalink` | 使用稳定、可读的英文路径；发布后原则上不修改。 |

## 可选字段

```yaml
modified_date: 2026-06-25 10:00:00 +0800
difficulty: intermediate
featured: true
cover_image: /assets/images/example/cover.png
update_note: "补充工具选型与图表选型建议。"
canonical_url:
```

| 字段 | 规则 |
| --- | --- |
| `modified_date` | 内容发生实质更新时填写，不用于修正错别字。 |
| `difficulty` | `beginner`、`intermediate` 或 `advanced`。 |
| `featured` | 仅用于少量代表性文章，首页最多展示 3 篇。 |
| `cover_image` | 站内绝对路径；图片应有明确内容价值。 |
| `update_note` | 一句话说明本次实质更新，显示在文章页。 |
| `canonical_url` | 内容首发于其他地址时填写。 |

## 文章类型

`article_type` 只能使用：

- `concept`：解释概念、原理或模型。
- `comparison`：比较多个方案并给出选择边界。
- `guide`：完成具体任务的操作指南。
- `troubleshooting`：定位和解决具体故障。
- `case-study`：包含背景、决策、实施和结果的实践复盘。
- `opinion`：有明确论点和证据的分析。

## 分类与标签

- 分类用于稳定的顶层信息架构，不随热点增加。
- 标签描述技术、平台和问题域。

## 文章评分

本站评分优先衡量读者实际收益，而不是文章是否宏大、体系化或像教程。具体问题、可搜索、能解决的短文可以高于泛泛而完整的长文。

| 维度 | 权重 | 判断重点 |
| --- | ---: | --- |
| 问题真实度 | 25% | 是否来自真实使用场景，是否有人会搜索这个问题。 |
| 解决确定性 | 25% | 是否给出明确、可验证、能执行的结论、步骤或选择。 |
| 长尾价值 | 20% | 未来几年是否仍可能被搜索、引用或复用。 |
| 内容稀缺性 | 15% | 是否比通用教程、官方文档或搜索结果更具体。 |
| 表达完整度 | 15% | 是否说明原因、边界、注意事项和验证方式。 |

评分处理规则：

- `8.0-10.0`：代表性文章，可以考虑设为 `featured: true`。
- `6.0-7.9`：可发布，但后续应继续压缩、补证据或补边界。
- `< 6.0`：不发布；要么重写，要么删除。
- 边界文章按读者实际收益判断，不按篇幅、写作投入或作者经历判断。

## 摘要、分享与搜索

- `description` 是文章在列表、搜索和社交平台上的默认摘要。
- 摘要不得使用“本文将介绍”等无信息量开头。
- 摘要应包含读者能获得的结果，而不是重复标题。
- 分享图片优先使用 `cover_image`，否则使用站点默认图片。

## 收藏短评

音乐与影视数据中的 `note` 是客观事实补充，不是个人短评、推荐语或审美判断。只有能补充现有字段之外的发行背景、改编来源、真实人物、成员身份、版本差异等事实时才填写；没有可靠且非重复的事实时置空。

音乐条目字段规则：

- `title` 和 `original_title` 使用所选版本的稳定歌名；中文歌不要因为 Apple Music、Spotify 等平台显示英文名就把英文名写入 `original_title`。
- `artist` 和 `artists` 使用稳定艺名或人名，不写括号合成显示名；艺名、中文名、本名、罗马字等关系写入 `_data/person_names.yml`。
- `date_label` 是内部日期来源，不直接等同于面向读者的发行介质；页面应根据 `date_label`、`release_title` 和歌曲名组合渲染为 `单曲`、`EP《名称》`、`专辑《名称》`、`原声《名称》`、`收录《名称》`、`节目《名称》`、`现场《名称》` 等短语。
- 当 `release_title` 与歌曲名明显不一致时，不要默认显示为 `单曲《release_title》`；除非该值明确是单曲包名，否则优先使用 `收录《release_title》` 或更具体的 `原声《release_title》`。
- 音乐页面该字段的展示标签使用 `来源`，避免把收录、节目、现场、创作年份都误称为发行。
- `tags` 只用于有筛选价值的分类，不重复标题、艺人、发行名或其他字段已经表达的词。
- 未经可靠外部事实确认的争议、相似、抄袭判断不写入 `note`。

影视条目字段规则：

- `premiere_date` 存储可核验日期；`premiere_note` 存储原始渠道、地点、电影节、电视台或平台，不直接承担面向读者的完整文案。
- 页面应把 `premiere_note` 渲染为三段式短语，例如 `日期 / 电影节 / Toronto International Film Festival`、`日期 / 首映 / Wembley Arena`、`日期 / 上映 / 中国大陆`、`日期 / 电视首播 / SBS`。
- 避免把“首播/上映/发行”这类动作词同时写在字段值和页面标签里造成重复。
- 影视页面该字段的展示标签使用 `播映`，由后续事件类型说明是电影节、首映、上映还是电视首播。

`note` 写作规则：

- 只写一句客观事实，避免主观形容、组织观点或个人观点。
- 音乐 `note` 可写版本差异、影视/节目关联、成员身份、改编来源或字段中没有表达的发行背景。
- 影视 `note` 可写改编来源、真实人物、制作背景、系列关系或字段中没有表达的类型事实。
- 禁止重复 `title`、`artist`、`release_title`、`date`、`date_label`、`tags`、`director`、`premiere_date`、`premiere_note`、`awards` 已经表达的信息；没有额外可靠事实时使用 `note: ""`。
- 禁止使用评价性表达，例如“动人”“有力量”“更贴近”“值得”“最有意思”“成立”“压得住”。

## 图片

- 替代文本必须描述图片传递的信息，不能只写文件名或“截图”。
- 正文图片使用 HTML `img`，显式填写原始 `width` 和 `height`，避免页面加载时跳动。
- 非首屏图片添加 `loading="lazy"`。
- 只在图片能补充步骤、结果或证据时使用图片；纯装饰图片不进入正文。

```html
<img
  src="{{ '/assets/images/example/result.png' | relative_url }}"
  alt="命令执行成功后的终端输出"
  width="1200"
  height="720"
  loading="lazy">
```

## 更新规则

实质更新时：

1. 保留原始 `date`。
2. 更新 `modified_date`。
3. 填写 `update_note`。
4. 检查步骤、截图、链接和结论是否仍然有效。

## 发布检查

- `content_score` 不低于 6.0，且 `score_basis` 同时说明价值和不足。
- 从读者问题或使用场景开始，不从作者经历开始。
- 在前 20% 内容中给出结论、步骤或选择建议。
- 删除可以被标题、摘要或链接替代的背景铺垫。
- 标题与摘要能够独立说明文章价值。
- 结论、步骤或建议有明确适用边界。
- 外部事实提供来源或验证方式。
- 代码和命令已执行验证。
- 图片包含替代文本。
- 文章已关联合理的分类和标签。
- 没有破坏已有永久链接。
